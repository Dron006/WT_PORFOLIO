Final Semester Project

Complaint Management System

Mission: Digital Grievance Orchestration

The Complaint Management System (CMS) is a comprehensive digital ecosystem engineered specifically for the administrative and student workflows at KLE Technological University. Our primary goal is to bridge the communication gap between the student body and university administration through a transparent, high-accountability platform.

System Objectives

Digital Complaint Portal

Provide students and faculty a simple platform to register complaints anytime, anywhere using their institutional email.

Escalation Mechanism

Automatically escalate complaints unresolved beyond 5 working days directly to the HOD for priority resolution.

User Authentication

Secure registration and login using institutional credentials with strictly defined role-based access control.

Analytics & Reporting

Maintain a transparent record and generate high-level reports filtered by status, category, and date range.

Complaint Assignment

Enable admins to delegate complaints to the relevant department and set priority: Low, Medium, or High.

Anonymous Complaints

Support anonymous submission to protect student identity while maintaining grievance accountability and integrity.

Real-Time Status Tracking

Ensure students receive email notifications and in-app alerts upon every status change in the complaint lifecycle.

Improve Experience

Reduce complaint resolution time and improve overall student and staff satisfaction at KLE Technological University.

Architecture

System Design & Documentation

Core Three-Tier Architecture

Tier 1 — Client Layer (React.js + Vite + Tailwind CSS)

The Client Layer forms the presentation tier of the CMS. Built entirely with React.js and bundled via Vite, it runs entirely inside the student or admin's browser without any server-side rendering. React components manage all UI state locally — complaint form inputs, dashboard filters, notification badge counts — using useState and useEffect hooks. Tailwind CSS drives the responsive layout, ensuring the interface adapts cleanly from desktop dashboards to mobile complaint submission forms. The client communicates with the backend exclusively through HTTP REST API calls, attaching a JWT Bearer token in every request header to prove the user's authenticated identity and role. This strict separation means the frontend has zero direct access to the database — it only receives what the API permits, which is a foundational security principle of client-server design.

Tier 2 — Application Layer (Node.js + Express.js)

The Application Layer is the intelligence of the system. The Express.js server runs on Node.js at PORT 5000 with CORS configured to accept only trusted frontend origins. Every incoming HTTP request passes through the JWT authentication middleware first — if the token is missing, expired, or tampered with, the request is rejected immediately before any business logic runs. Valid requests then reach the appropriate route handler, where the core logic executes: creating complaint records with unique IDs, assigning workers, triggering the auto-escalation scheduler for complaints older than five working days, sending real-time notifications through the Notification Service, and generating PDF or Excel reports on demand. The application layer acts as a strict gatekeeper — students cannot access admin routes, workers cannot see other departments' complaints, and HODs only see escalated cases. This role-based logic lives entirely in this tier, protecting data integrity from both the client and the database sides.

Tier 3 — Data Layer (MongoDB + Mongoose ODM)

The Data Layer provides persistent, structured storage through MongoDB's complaint_cms database. Mongoose ODM defines strict schemas for every collection — students, faculty, admins, workers, complaints, notifications, and files — ensuring that malformed or incomplete documents are rejected before they ever reach the database. Complaint records store their full lifecycle: submission timestamp, category, description, assigned worker reference, priority level, current status, anonymous flag, and escalation state. This design means the entire history of every complaint is preserved permanently, enabling admins to generate accurate analytical reports filtered by date range, category, or department at any point in time.

Why This Architecture Fits the CMS

The three-tier separation delivers practical advantages specifically suited to an institutional grievance system. Independent scaling means that if complaint submissions spike during exam season, only the application server needs horizontal scaling — the React frontend served via CDN and the MongoDB Atlas cluster remain unaffected. Security is enforced at the application boundary, not the client, so there is no risk of a student manipulating browser code to access admin data. The clean API contract between tiers also allowed the team to develop and test the React frontend and the Express backend in parallel using mocked API responses, which accelerated the development timeline significantly under the academic sprint schedule.

Sequence Diagram 1: Complaint Submission Flow

This diagram traces the complete lifecycle of a complaint from the moment a student clicks Submit to the point their dashboard refreshes with the new complaint card. The flow begins entirely on the client side — the React frontend collects the form data (category, description, anonymous toggle) and packages it as a JSON body alongside the stored Bearer token in the Authorization header, then fires a POST /api/complaints request to the Express API. The Express server does not touch any database immediately — the JWT Middleware intercepts the request first, extracts the token, verifies its signature against the server's JWT_SECRET, and checks that the decoded role is either student or faculty. Only after that role check passes does the middleware call next(), forwarding the request to the complaint route handler. The Complaint Model then executes Complaint.create() with the validated payload, generating a unique complaint ID in the format CMS-2025-001 and writing the document to MongoDB with an initial status of Pending. Once the database confirms the save, the Notification Service is triggered asynchronously — it dispatches a real-time in-app alert and an email notification to the student confirming successful submission. Finally, the API returns a 201 Created response containing the complaint ID back to the React frontend, which uses it to immediately refresh the student dashboard so the new complaint card appears without requiring a full page reload. This unidirectional, layered flow ensures that authentication always precedes data access and that no complaint is created without a verified, role-confirmed identity behind it.

Sequence Diagram 2: Authentication & Token Issuance Flow

This diagram describes how a user of any role — student, faculty, admin, HOD, or worker — is authenticated into the system using a single unified login endpoint. The user enters their institutional KLE email and password in the React frontend login form, which packages them into a JSON body and sends a POST /api/login request to the Express /login route. Because the CMS has five distinct user roles stored across five separate MongoDB collections, the User Model runs a findOne({ email }) query across all role collections simultaneously — if the email belongs to a student, the student document is returned; if it belongs to an admin, the admin document comes back instead. This multi-role lookup pattern means the system needs only one login endpoint instead of five separate ones, reducing API surface area and simplifying the client-side logic. Once the matching user document is retrieved with its stored password_hash, the password the user typed is passed to bcrypt.compare(), which rehashes it with the same salt and checks it against the stored hash — this comparison is deliberately slow by design (SALT_ROUNDS=12) to make brute-force attacks computationally expensive. If bcrypt.compare() returns true, the Express server calls jwt.sign() with the user's ID, email, and role embedded in the payload, signed with the JWT_SECRET, and set to expire after 7 days. The resulting token is sent back in a 200 OK response alongside the user's role. The React frontend stores this token in memory and uses it to redirect the user to their role-specific dashboard — the student sees their complaints, the admin sees the full management panel, and the HOD sees only escalated cases. From this point forward, every API request the user makes will carry this token, and the JWT Middleware will use it to verify their identity without ever querying the database again for authentication purposes.

Implementation

High-Performance Engineering

Technical Stack Integration

The CMS project is built on a modern MERN (MongoDB, Express, React, Node.js) foundation, optimized for institutional scalability. My implementation leverages a Tier-3 Client-Server architecture where the frontend (Client) communicates with the Node.js server via a RESTful API. This approach ensures that university data is never exposed directly and all operations are mediated by secure controllers.

Security (JWT & Bcrypt)

We utilize Bcrypt for one-way salting and hashing of passwords, ensuring zero plaintext storage. Session management is handled via JSON Web Tokens (JWT), enabling stateless, role-based authorization across all API endpoints.

Data (Base64 Imaging)

To simplify evidence management, complaint images are converted to Base64 strings. This allows us to store visual evidence directly within MongoDB collections, eliminating the need for external S3 buckets for initial deployment.

State (React Hooks)

The frontend leverages Advanced React Hooks (useState, useEffect, useMemo) to manage complex multi-role dashboards and real-time form validations without performance bottlenecks.

The Engineering Challenges

Binary Storage & Portability

One of the most significant hurdles was managing student-uploaded evidence without a dedicated file server. I solved this by implementing Base64 Data Streams, encoding image buffers into strings that could be natively stored in MongoDB. This ensured the entire project remained portable and easy to deploy on institutional cloud tiers.

Complex Role-Based Routing

Synchronizing the view for five distinct user types (Student, faculty, Admin, HOD, Worker) while maintaining a single SPA route required advanced React lifecycle management. Ensuring that an HOD sees escalated complaints while a Worker only sees assigned ones was a major test of data filtering and conditional rendering logic.
